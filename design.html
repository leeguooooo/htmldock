设计原则：动态上传单一来源

所有用户可见的"文档"内容必须经过 CLI / Skill 的上传通道进入平台。索引、导航、搜索结果页都是 Worker 实时渲染——它们消费 D1，但本身不是被上传的 HTML。这条原则约束了：没有"在网页里直接编辑文档"的功能（一期不做）；没有"上传到 R2 但不在 D1 登记"的旁路。

§1 整体架构

两条独立访问路径：团队成员凭 Lark session cookie 看私有文档；外部通过 /p/<token> 看公开分享，Worker 自验签，不依赖 session。

安全边界：上传的 HTML 绝不能和应用/API session cookie 同源直出。应用/API 使用 app origin；文档内容使用独立 content origin。/d/<doc-id> 先在应用 origin 校验 session，再签发短期 view token 跳转到 content origin；content origin 不带 session cookie、不开私有 API，只返回极简阅读页外壳，并在其中加载原始 HTML。公开分享 /p/<token> 也走 content origin，并默认返回 Referrer-Policy: no-referrer 与 X-Robots-Tag: noindex。

-- D1 FTS5 全文索引（content-less，rowid 与 docs.id 对齐）
CREATE VIRTUAL TABLE docs_fts USING fts5(
  title, body_text, content=''
);
FTS5 同步说明：不使用 D1 trigger 调不存在的 extractBodyText()。纯文本提取（剥 HTML 标签、解 entity、压缩空白）在 Worker 端完成；docs 与 docs_fts 必须在同一个 D1 batch / transaction 语义里写入、更新或删除，保证 docs_fts.rowid === docs.id。

R2 key 规则（v0.5 改用 team/project slug）

v0.4 用 git host/owner/repo 拼 key，会把 GitHub 用户名暴露进 R2 路径。v0.5 改为基于 team-slug 与 project-slug：

r2_key = "t/<team-slug>/<project-slug>/<path>"

例：t/acme-infra/cherry/auth/login-flow.html

R2 key 与平台 URL 形态一致。同样 Q1 选了原地覆盖，key 就是路径本身。删除文档时，Worker 同步删除该 R2 对象；删除项目时，先 list 该项目 R2 前缀下所有对象并批量删除。slug 一旦创建不允许改名（避免 R2 大规模迁移）；改名只能"删旧建新"。

Q1 已落定：R2 原地覆盖 + docs 仅保存当前 sha256，不引入 doc_revisions。明确不承诺历史、回滚、immutable share，但保留 _rev/<doc_id>/... 命名空间作为后续扩展点。

§3 路径坐标算法

CLI 从 .htmldock.toml、git remote -v、git rev-parse --show-toplevel 和 HTML 文件路径推导上传坐标 (team_slug, project_slug, source_path, path) + git 元数据 (host, owner, repo)。team_slug 来自 .htmldock.toml（必填），不从 git remote 推导；project_slug 默认 = git repo basename，可在 .htmldock.toml 覆盖；git host/owner/repo 仅作为元数据，用来匹配现有 platform project，并通过 UNIQUE(git_host, git_owner, git_repo) 防止同一 git repo 注册到两个 team。

解析伪代码

function deriveCoordinates(filePath, options):
  root = git("rev-parse --show-toplevel", cwd=dirname(filePath))
  if root is empty:
    error("HTML file must be inside a git repository")

  remotes = parseGitRemoteV(git("remote -v", cwd=root))
  remote  = chooseRemote(remotes, options.remote)
  if remote is empty:
    if options.allowLocalNamespace:
      host  = "local@" + hostname()
      owner = username()
      repo  = basename(root)
    else:
      error("No git remote found; pass --allow-local to use local namespace")
  else:
    parsed = parseRemoteUrl(remote.fetchUrl)
    host   = normalizeHost(parsed.host)
    owner  = trimSlashes(parsed.ownerPath)
    repo   = stripSuffix(parsed.repo, ".git")

  relFile = relativePath(root, absolutePath(filePath))
  if relFile startsWith "..":
    error("File is outside repository root")

  projConf = readTomlIfExists(root + "/.htmldock.toml")
  moduleRoot = normalizeRelativeDir(projConf.module_root or "")
  if moduleRoot and not relFile.startsWith(moduleRoot + "/"):
    error("file_not_in_module_root")

  displayPath = moduleRoot ? relFile.removePrefix(moduleRoot + "/") : relFile

  // 模块树深度校验（见 §5）：校验展示路径，不校验 source_path
  segments = displayPath.split("/")
  if segments.length > 3:
    error("path_too_deep: max 3 segments including filename, got " + segments.length)

  team_slug    = projConf.team
                 or error("team is required in .htmldock.toml")
  project_slug = projConf.project_slug or basename(root)

  return {
    team_slug:    team_slug,
    project_slug: project_slug,
    host:         host,
    owner:        owner,
    repo:         repo,
    source_path:  relFile,
    path:         displayPath
  }

push 内部步骤

  1. 校验文件存在且后缀为 .html。
  2. 跑 git rev-parse --show-toplevel 找仓库根；找不到则拒绝（非 git 文件）。
  3. 读项目级 .htmldock.toml（见 §5）。若不存在 → 按 default_project_sync 决定（默认 opt-in，未开启则 exit 0 并提示 Run "htmldock init" to enable sync）。team 必填，缺失则拒绝。
  4. 跑 git remote -v，按 §3 算 (team_slug, project_slug, host, owner, repo, source_path, path)。
  5. 调 POST /api/projects/ensure：在 team_slug 下查找/创建 project_slug；若 git remote 已经绑到别的 team/project，服务端返回 project_conflict。校验当前用户是该 team 成员，否则 403。
  6. 校验展示 path 段数 ≤ 3；若配置 module_root，先剥离后再校验。超过 → 报 path_too_deep 并提示压扁建议。
  7. 从 HTML 抽 <title>，回退到 filename。
  8. 算 sha256。
  9. GET /api/docs/check?project=…&path=…&sha256=…：若已存在且哈希一致 → 打印 Already up to date 并 exit 0，不更新 updated_at。
  10. POST /api/docs multipart：file + metadata JSON。
  11. 打印 ✓ Published: https://htmldock.example.com/d/<doc-id>。

§5 Skill / Hook 机制

本节定义"Generate = Upload"：当 Claude Code 写入 HTML 文件时，PostToolUse Hook 调用 htmldock push；并通过 .htmldock.toml 实现项目级 opt-in 开关；同时给 agent 一个"主动选模块路径"的指令骨架。

项目级配置：.htmldock.toml

位置：每个想接入平台的 repo 根目录。可由 htmldock init 交互生成。

# <repo-root>/.htmldock.toml
team           = "acme-infra"          # 必填。该 repo 归属的 team slug
project_slug   = "cherry"              # 可选；默认 = git repo basename
sync           = true                  # 必填。false / 缺失文件 = 不同步
visibility     = "team"                # team | public-allowed | private-strict
default_owner  = "leo"                 # push 时不指定 --owner 的 fallback
module_root    = ""                    # 仅这个子目录下生成的 HTML 才被认为是"成品"，留空表示整个 repo
ignore         = ["draft/**", "scratch/*.html"]
auto_module    = true                  # Skill 是否自动建模块树（见下）
字段类型缺省说明
teamstring必填该 repo 归属的 team slug。team 必须已存在；用户必须是该 team 成员。不可由 git remote 推导，必须显式声明。
project_slugstringgit repo basename覆盖默认 project slug。仅在创建项目时生效；项目一旦创建不可改名。
syncbool必填本仓库是否启用同步。false 或文件缺失都不会上传（除 --force）。
visibilityenumteamteam: 团队内可读；public-allowed: 允许生成公开链接；private-strict: 仅作者本人可读，禁止 share。
default_ownerstring当前 Lark 用户push 时未传 --owner 的归属人。
module_rootstring"""成品文档"目录。设为 "docs" 后，scratch/notes.html 不会上传。
ignorestring[][]glob 列表，叠加在全局 ignore_patterns 之上。
auto_modulebooltrueSkill 是否在生成 HTML 前主动选模块路径（见下）。关闭则 agent 把 HTML 平铺到 repo 根。

opt-in vs opt-out

方案含义优点风险推荐
opt-in 仅存在 .htmldock.toml 且 sync = true 的 repo 才上传 不会误传别人的代码 / 实验 / 私库；新 repo 默认安全；和 git init 后第一次 commit 一样需要显式动作。 稍多一步 htmldock init。 推荐
opt-out 所有 git 仓库默认同步，.htmldock.toml 写 sync = false 来关 极简：写完即上。 容易把含密钥 / 内部敏感内容的 repo 误传；/Users/leo/github.com/ 下 100+ 仓库多数是别人的代码，全开同步噪音很大。 —
Q2 已落定：默认 opt-in。default_project_sync = "ask" 只允许在前台 TTY 的手动 CLI 场景使用；Hook 路径必须把 ask 当作 off，避免非交互环境挂住或吞掉提示，造成用户以为已同步但实际没有上传。

模块树规则（深度 ≤ 3）

所有上传文档的展示 path 段数（含文件名）不得超过 3。若配置 module_root，先从 repo 相对路径中剥离 module_root，再做深度校验并写入 docs.path；未剥离的真实 repo 路径写入 docs.source_path。服务端 API 强校验，超出返回 400 path_too_deep。

✓ 合规示例

✗ 不合规（深度 > 3）

语义动机：深度 ≤ 3 是为了让索引页项目树永远 1 屏可见，并强迫作者用文件名而非路径表达细分（oauth/lark-integration.html 比 oauth/lark/integration.html 信息量等价但更扁）。

htmldock Skill（~/.claude/skills/htmldock/SKILL.md）

# htmldock

Use this skill whenever you create an HTML technical document
(spec, plan, review, runbook, design doc) inside a project that has
`.htmldock.toml` with `sync = true`.

## Before writing the HTML

1. Run `htmldock list --project=current --tree` to see the existing
   module tree for this project.
2. Decide where the new doc belongs:
   - If the doc's topic matches an existing top-level module (e.g.
     `auth/`, `billing/`, `infra/`), **reuse that prefix**.
   - Don't create synonyms (`authentication/` when `auth/` already
     exists; `infrastructure/` when `infra/` exists).
   - If no module fits, choose the shortest meaningful prefix
     (1-2 words, kebab-case, matching the project's existing
     naming style).
3. Keep depth ≤ 3 levels **including the filename**:
   - 1 level: `release-notes.html`
   - 2 levels: `auth/login-flow.html`
   - 3 levels: `auth/oauth/lark-integration.html`
   - **Never** 4 levels — the server rejects with `path_too_deep`.
4. Don't manufacture hierarchy. If 1 level is enough, use 1 level.

## After writing the HTML

The PostToolUse hook auto-uploads. If it fails, you'll see a queue
entry; the next `htmldock push` drains it. No need to invoke push
yourself unless the user explicitly asks "publish this now".

## When the user says "publish this"

Run `htmldock push <filepath>`. Do not edit the HTML unless
explicitly asked. Report the printed URL back.
Skill 与 Hook 的关系：Hook 是"自动"的，针对所有写入的 HTML；Skill 是"显式"的，针对用户的自然语言请求。两者共用 CLI htmldock push，互不依赖；Hook 失败不会触发 Skill，Skill 调用也不依赖 Hook 是否安装。

§6 认证流程（Lark OAuth only）

MVP 只实现 Lark OAuth。浏览器使用 session cookie，CLI 使用 Personal Access Token（PAT）。Cloudflare Access SSO 保持延期，不进入本节实现范围。

Origin / cookie 边界

Origin职责Cookie / API 规则
app.htmldock.example.com登录、索引页、私有 API、签发 view token。设置 host-only session cookie：HttpOnly; Secure; SameSite=Lax，不设置 Domain=.htmldock.example.com。
content.htmldockusercontent.example只按短期 view token 或公开 share token 返回独立阅读页，并加载 R2 HTML。不设置 session cookie，不开放私有 API；阅读页外壳极简，原 HTML 保留自己的样式；响应设置隔离与反索引头。

私有文档访问流程：浏览器访问 GET /d/<doc-id>（app origin）→ Worker 校验 Lark session 与文档权限 → 生成 60 秒 view token（绑定 doc_id、user_id、nonce）→ 302 到 content origin 的 /v/<view-token> → content Worker 校验 token 后返回独立阅读页。阅读页最多包含一个小导航条，并把原始 HTML 作为主体内容加载；平台不向作者 HTML 注入全局样式，也不要求作者 HTML 遵守平台 CSS 规范。

OAuth state / redirect 规则

  • state 必须随机生成、只存哈希、写入 oauth_states，TTL 10 分钟，callback 成功后标记 consumed_at，不可复用。
  • Browser flow 的 redirect_uri 必须精确匹配部署环境 allowlist，如 https://app.htmldock.example.com/api/auth/lark/callback。
  • CLI flow 只允许 local_cb 为 http://127.0.0.1:<port>/save 或 http://localhost:<port>/save，禁止任意外部 callback。
  • Worker 兑换 code 前先校验 state、用途、TTL、redirect allowlist；任何失败返回 400 并不创建 session 或 PAT。

PAT 规则

PAT 不再存放在 users.pat_hash 单字段里。MVP 使用 personal_access_tokens 表支持多设备、撤销、过期、last_used 与 scope。CLI token 明文只在创建时显示一次，本地写入 ~/.config/htmldock/config.toml；服务端只保存 hash 与短 prefix。上传接口要求 docs:write，分享接口要求 share:write。

Public share flow（no auth）: GET /p/<token> 进入 content origin 后，查 shares 表：要求 token 存在、未 revoked、未过期；同时校验 docs.visibility 为 public-allowed。通过 doc_id 找到 docs.r2_key，返回同样的独立阅读页外壳和原始 HTML 主体，响应包含 Referrer-Policy: no-referrer、X-Robots-Tag: noindex。该路径不要求 session cookie 或 PAT。

Share token 规则

  • token 使用 128-bit 以上随机熵，base64url 编码，长度不少于 22 字符；只在 URL 中展示，不可由 doc_id 推导。
  • MVP 默认 TTL 为 30 天；显式 --ttl 0 才创建永久链接。永久链接仍可 revoke。
  • private-strict 禁止创建 share；team 只允许团队内访问；只有 public-allowed 可公开分享。

§7 阅读页 / 索引页 UX

All index pages are Worker-rendered from D1. No HTML is uploaded to represent the index — the Worker generates it on every request.

阅读页：独立、极简、不约束 HTML 风格

用户点击 /d/<doc-id> 或 /p/<token> 后看到的是单独的文档阅读页，不进入 dashboard 或三栏工作台。阅读页只允许一个很小的导航条，用于返回索引、显示标题/项目、复制链接或打开分享动作；导航条不得承载搜索、侧栏、推荐、评论、目录树等复杂功能。

HTML 文档是主体，平台不为作者 HTML 制定视觉规范，不注入 reset、排版、颜色、间距、字体等全局限制。每份 HTML 可以有自己的完整风格；平台只负责上传、鉴权、索引、搜索和分享。

content page
  small nav: Back · title · Share/Open raw
  document body: original uploaded HTML, style owned by the document
