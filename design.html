设计原则：动态上传单一来源

所有用户可见的"文档"内容必须经过 CLI / Skill 的上传通道进入平台。索引、导航、搜索结果页都是 Worker 实时渲染——它们消费 D1，但本身不是被上传的 HTML。这条原则约束了：没有"在网页里直接编辑文档"的功能（一期不做）；没有"上传到 R2 但不在 D1 登记"的旁路。

§1 整体架构

两条独立访问路径：团队成员凭 Lark session cookie 看私有文档；外部通过 /p/<token> 看公开分享，Worker 自验签，不依赖 session。

§2 数据模型

D1 表结构（SQL）

-- 项目坐标（来自 git remote 解析）
projects (
  id            INTEGER PRIMARY KEY,
  host          TEXT NOT NULL,      -- github.com / gitlab.com / 内网
  owner         TEXT NOT NULL,      -- leeguoo / org
  repo          TEXT NOT NULL,
  display_name  TEXT,
  created_at    INTEGER,
  UNIQUE(host, owner, repo)
)

-- 文档主表
docs (
  id            INTEGER PRIMARY KEY,
  project_id    INTEGER REFERENCES projects(id),
  path          TEXT NOT NULL,      -- 项目内相对子路径，最多 3 段（含文件名），如 auth/login-flow.html
  title         TEXT,               -- 解析 <title> 提取
  r2_key        TEXT NOT NULL,      -- R2 对象 key（与 path 一一对应，无 query 字符）
  owner_user_id INTEGER REFERENCES users(id),
  size_bytes    INTEGER,
  sha256        TEXT,               -- 当前内容哈希，幂等性的依据
  created_at    INTEGER,
  updated_at    INTEGER,
  UNIQUE(project_id, path)
)

users (
  id            INTEGER PRIMARY KEY,
  lark_open_id  TEXT UNIQUE,
  email         TEXT,
  name          TEXT,
  avatar_url    TEXT,
  pat_hash      TEXT,               -- CLI 用 PAT（SHA256 哈希存储）
  created_at    INTEGER
)

shares (
  token         TEXT PRIMARY KEY,   -- 22 字节 base64url，不可猜测
  doc_id        INTEGER REFERENCES docs(id),
  created_by    INTEGER REFERENCES users(id),
  expires_at    INTEGER,            -- NULL = 永久
  revoked       INTEGER DEFAULT 0,
  created_at    INTEGER
)

-- D1 FTS5 全文索引（content-less，rowid 与 docs.id 对齐）
CREATE VIRTUAL TABLE docs_fts USING fts5(
  title, body_text, content=''
);

-- 触发器：保证 docs_fts.rowid === docs.id
CREATE TRIGGER docs_ai AFTER INSERT ON docs BEGIN
  INSERT INTO docs_fts(rowid, title, body_text)
  VALUES (new.id, new.title, extractBodyText(new.r2_key));
END;
CREATE TRIGGER docs_au AFTER UPDATE OF title, sha256 ON docs BEGIN
  INSERT INTO docs_fts(docs_fts, rowid, title, body_text)
  VALUES ('delete', old.id, old.title, '');
  INSERT INTO docs_fts(rowid, title, body_text)
  VALUES (new.id, new.title, extractBodyText(new.r2_key));
END;
CREATE TRIGGER docs_ad AFTER DELETE ON docs BEGIN
  INSERT INTO docs_fts(docs_fts, rowid, title, body_text)
  VALUES ('delete', old.id, old.title, '');
END;
FTS5 同步说明：extractBodyText() 不是真的 SQL 函数；纯文本提取（剥 HTML 标签、解 entity、压缩空白）在 Worker 端完成后随 INSERT/UPDATE 一并写入。触发器只是表达 rowid 一一对应的语义；实际写入由 Worker 在事务里做。

R2 key 规则（修订自 v0.2）

v0.2 的 <path>?v=<sha> 把 ? 放进了 key，会和 URL query 语义混淆。v0.3 简化为：

r2_key = "<host>/<owner>/<repo>/<path>"

例：github.com/leeguoo/cherry/auth/login-flow.html

由于 Q1 选了"原地覆盖、不留历史"（Option B），key 就是路径本身，不再带版本前缀。日后若引入 doc_revisions，可在历史 key 上挂时间戳前缀如 _rev/<doc_id>/<ts>/...，跟当前 key 不冲突。

Q1 已落定（v0.2 推荐）：R2 原地覆盖 + docs 仅保存当前 sha256，不引入 doc_revisions。理由：HTML 文档更新频率低、回滚价值低、R2 按对象计费、D1 schema 越简越好。

§3 路径坐标算法

CLI 从 git remote -v、git rev-parse --show-toplevel 和 HTML 文件路径推导上传坐标 (host, owner, repo, sub_path)。其中 host/owner/repo 定义项目，sub_path 定义该 HTML 文件在项目内的位置；作者身份只作为 metadata，不参与路径坐标。

输入模式与解析结果

https://github.com/leeguoo/cherry.gitgithub.comleeguoocherry标准 HTTPS URL。用 URL parser，不字符串硬切。

解析伪代码

function deriveCoordinates(filePath, options):
  root = git("rev-parse --show-toplevel", cwd=dirname(filePath))
  if root is empty:
    error("HTML file must be inside a git repository")

  remotes = parseGitRemoteV(git("remote -v", cwd=root))
  remote  = chooseRemote(remotes, options.remote)
  if remote is empty:
    if options.allowLocalNamespace:
      host  = "local@" + hostname()
      owner = username()
      repo  = basename(root)
    else:
      error("No git remote found; pass --allow-local to use local namespace")
  else:
    parsed = parseRemoteUrl(remote.fetchUrl)
    host   = normalizeHost(parsed.host)
    owner  = trimSlashes(parsed.ownerPath)
    repo   = stripSuffix(parsed.repo, ".git")

  relFile = relativePath(root, absolutePath(filePath))
  if relFile startsWith "..":
    error("File is outside repository root")

  // 模块树深度校验（见 §5）
  segments = relFile.split("/")
  if segments.length > 3:
    error("path_too_deep: max 3 segments including filename, got " + segments.length)

  return {
    host:     host,
    owner:    owner,
    repo:     repo,
    sub_path: dirname(relFile),
    path:     relFile
  }

§4 CLI 设计（htmldock, Bun single-binary）

htmldock 是 Bun 编译出的单文件 CLI。它只负责本地文件校验、Git 坐标解析、项目级配置读取、认证 token 管理和 API 调用——不嵌 HTML 编辑器。

命令表

init无在当前 repo 根写入 .htmldock.toml（带交互问答：sync? visibility? module_root?）。--yes（一路默认）

  "*-draft.html", "tmp-*.html"
]

push 内部步骤

  1. 校验文件存在且后缀为 .html。
  2. 跑 git rev-parse --show-toplevel 找仓库根；找不到则拒绝（非 git 文件）。
  3. 读项目级 .htmldock.toml（见 §5）。若不存在 → 按 default_project_sync 决定（默认 opt-in，未开启则 exit 0 并提示 Run "htmldock init" to enable sync）。
  4. 跑 git remote -v，按 §3 算 (host, owner, repo, sub_path, path)。
  5. 校验 path 段数 ≤ 3；超过 → 报 path_too_deep 并提示压扁建议。
  6. 从 HTML 抽 <title>，回退到 filename。
  7. 算 sha256。
  8. GET /api/docs/check?project=…&path=…&sha256=…：若已存在且哈希一致 → 打印 Already up to date 并 exit 0，不更新 updated_at。
  9. POST /api/docs multipart：file + metadata JSON。
  10. 打印 ✓ Published: https://htmldock.example.com/d/<doc-id>。

全局 flags

Flag行为
--force绕过项目级 .htmldock.toml 的 sync = false，强行单次上传。不会写入项目配置。
--dry-run打印将解析出的坐标、title、sha256、目标 server，不发起网络请求。

§5 Skill / Hook 机制

本节定义"Generate = Upload"：当 Claude Code 写入 HTML 文件时，PostToolUse Hook 调用 htmldock push；并通过 .htmldock.toml 实现项目级 opt-in 开关；同时给 agent 一个"主动选模块路径"的指令骨架。

项目级配置：.htmldock.toml

位置：每个想接入平台的 repo 根目录。可由 htmldock init 交互生成。

# <repo-root>/.htmldock.toml
sync           = true                  # 必填。false / 缺失文件 = 不同步
visibility     = "team"                # team | public-allowed | private-strict
default_owner  = "leo"                 # push 时不指定 --owner 的 fallback
module_root    = ""                    # 仅这个子目录下生成的 HTML 才被认为是"成品"，留空表示整个 repo
ignore         = ["draft/**", "scratch/*.html"]
auto_module    = true                  # Skill 是否自动建模块树（见下）
字段类型缺省说明
syncbool必填本仓库是否启用同步。false 或文件缺失都不会上传（除 --force）。
visibilityenumteamteam: 团队内可读；public-allowed: 允许生成公开链接；private-strict: 仅作者本人可读，禁止 share。
default_ownerstring当前 Lark 用户push 时未传 --owner 的归属人。
module_rootstring"""成品文档"目录。设为 "docs" 后，scratch/notes.html 不会上传。
ignorestring[][]glob 列表，叠加在全局 ignore_patterns 之上。
auto_modulebooltrueSkill 是否在生成 HTML 前主动选模块路径（见下）。关闭则 agent 把 HTML 平铺到 repo 根。

opt-in vs opt-out

方案含义优点风险推荐
opt-in 仅存在 .htmldock.toml 且 sync = true 的 repo 才上传 不会误传别人的代码 / 实验 / 私库；新 repo 默认安全；和 git init 后第一次 commit 一样需要显式动作。 稍多一步 htmldock init。 推荐
opt-out 所有 git 仓库默认同步，.htmldock.toml 写 sync = false 来关 极简：写完即上。 容易把含密钥 / 内部敏感内容的 repo 误传；/Users/leo/github.com/ 下 100+ 仓库多数是别人的代码，全开同步噪音很大。 —

Q2：opt-in 还是 opt-out？

  • 我建议默认 opt-in。是否同意？或希望走 opt-out？
  • 全局 default_project_sync 是否要支持 "ask"（首次 push 时弹问）？

模块树规则（深度 ≤ 3）

所有上传文档的 path 段数（含文件名）不得超过 3。服务端 API 强校验，超出返回 400 path_too_deep。

✓ 合规示例
auth-redesign.html
auth/login-flow.html
auth/oauth/lark-integration.html
billing/invoice-export.html
infra/observability/dashboard.html
✗ 不合规（深度 > 3）
auth/oauth/lark/v2/proposal.html
docs/team/qa/runbook/incident-1024.html
src/api/v2/handlers/upload.html
语义动机：深度 ≤ 3 是为了让索引页项目树永远 1 屏可见，并强迫作者用文件名而非路径表达细分（oauth/lark-integration.html 比 oauth/lark/integration.html 信息量等价但更扁）。

htmldock-hook 伪代码

input = readStdinJson()
path  = extractWrittenFilePath(input)
if path is empty or not path.endsWith(".html"):
  exit 0

repoRoot = run("git rev-parse --show-toplevel", cwd=dirname(path))
if repoRoot failed:
  exit 0

# opt-in 守门
projConf = readTomlIfExists(repoRoot + "/.htmldock.toml")
if projConf is None:
  switch globalConfig.default_project_sync:
    case "off":  exit 0
    case "on":   projConf = {sync: true}
    case "ask":  prompt user once, cache decision
elif not projConf.sync:
  exit 0

# ignore 列表
ignore = globalConfig.ignore_patterns + (projConf.ignore or [])
if projConf.module_root and not pathRelativeTo(repoRoot, path).startsWith(projConf.module_root):
  exit 0    # 不在"成品目录"
if matchesAny(pathRelativeTo(repoRoot, path), ignore):
  exit 0

# 上传
result = run("htmldock push --quiet " + shellQuote(path))
if result failed:
  appendJsonl("~/.local/share/htmldock/queue.jsonl", {
    ts: nowUnix(), path: path, error: result.stderr
  })
exit 0   # 永不打断 Claude Code 主流程

失败队列与 replay

Hook 失败一律静默；失败条目追加到 ~/.local/share/htmldock/queue.jsonl。下一次任意 htmldock push 成功后，CLI 顺手 drain 该队列：逐条重试，成功的删除，失败的保留并更新最近错误。

htmldock Skill（~/.claude/skills/htmldock/SKILL.md）

# htmldock

Use this skill whenever you create an HTML technical document
(spec, plan, review, runbook, design doc) inside a project that has
`.htmldock.toml` with `sync = true`.

## Before writing the HTML

1. Run `htmldock list --project=current --tree` to see the existing
   module tree for this project.
2. Decide where the new doc belongs:
   - If the doc's topic matches an existing top-level module (e.g.
     `auth/`, `billing/`, `infra/`), **reuse that prefix**.
   - Don't create synonyms (`authentication/` when `auth/` already
     exists; `infrastructure/` when `infra/` exists).
   - If no module fits, choose the shortest meaningful prefix
     (1-2 words, kebab-case, matching the project's existing
     naming style).
3. Keep depth ≤ 3 levels **including the filename**:
   - 1 level: `release-notes.html`
   - 2 levels: `auth/login-flow.html`
   - 3 levels: `auth/oauth/lark-integration.html`
   - **Never** 4 levels — the server rejects with `path_too_deep`.
4. Don't manufacture hierarchy. If 1 level is enough, use 1 level.

## After writing the HTML

The PostToolUse hook auto-uploads. If it fails, you'll see a queue
entry; the next `htmldock push` drains it. No need to invoke push
yourself unless the user explicitly asks "publish this now".

## When the user says "publish this"

Run `htmldock push <filepath>`. Do not edit the HTML unless
explicitly asked. Report the printed URL back.
Skill 与 Hook 的关系：Hook 是"自动"的，针对所有写入的 HTML；Skill 是"显式"的，针对用户的自然语言请求。两者共用 CLI htmldock push，互不依赖；Hook 失败不会触发 Skill，Skill 调用也不依赖 Hook 是否安装。

§8 错误处理与测试

幂等性：如果 (project_id, path, sha256) 已存在于 docs，跳过 R2 write，返回 200 和已有 doc URL，不更新 updated_at。理由：sha256 一致代表内容相同，timestamp churn 浪费 D1 写、污染 recent updates。

Retry strategy

AttemptWait Before RetryAction
1st fail0sImmediate retry。
2nd fail2sShort delay retry。
3rd fail8sFinal network retry。
4th fail—写 queue.jsonl，exit with error message。
